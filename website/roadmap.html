Official roadmap

The project follows a strict construction order. Later extensions and the anticipation engine cannot replace the Core, schemas, tools and interoperability proof that must come first.

Open the master planTrack issues
Mandatory order: Core model → Schema family → Unified validator and comparator → Migration → JSON/CSV compiler → Review interface → Value proof → Extensions → Anticipation engine.
0Complete

Definition

Nine domain profiles, the normative ten-block Core object model, module boundaries and architectural safeguards.

  • Minimum envelope: aduc + resource + structure
  • One owning module per normative fact
  • Twenty-five architecture counterexamples checked in CI
1Complete

Modular schema family

Fourteen local Draft 2020-12 schemas, eleven complete valid contracts, fifteen invalid contracts and one combined schema/architecture validator.

  • No remote schema retrieval
  • Closed Core objects and explicit extensions
  • The complete ten-block example validates locally
2Complete

Unified validation and comparison

One stable Core error catalogue, one validation pipeline and one deterministic comparator across semantics, units, time, identity, provenance, uncertainty, relations and policy.

  • Existing domain evaluators remain authoritative and are called through Core adapters
  • Change type, compatibility classification and semantic assessment are separate
  • Unknown, contested, deprecated and prohibited dimensions remain visible
3Next active task

Semantic-profile migration

Migrate the standalone mapping profile into complete Core contracts, then import JSON and CSV only after the review-tax and adoption gates are ready.

4Not started

Review interface

Show unknown fields, low-confidence mappings, ambiguous units, time conflicts, identity candidates, provenance gaps, uncertainty assumptions and relation or policy conflicts.

5Harness ready

Value and interoperability demonstration

Manual and assisted mapping benchmarks, two incompatible sources, two independent AI consumers and reproducible comparison evidence.

  • Provider-neutral harness exists
  • A qualifying value benchmark and external runs remain absent
6Blocked by proof

Extensions

Dataset or Live Data extensions begin only after the Core passes the value and interoperability gate.

7Future

Situation & Action extension

Represent situations, possible developments, actions, constraints and evidence without turning the Core into a decision engine.

8Future application

Anticipation engine

A separately governed application using ADUC Core plus optional extensions.

9Future ecosystem

Registry, connectors and governance

Public and private registries, community extensions, certification, support and independent governance.

Version 0.1

Completion requires reproducible evidence.

The project will not call v0.1 complete because a website or a long document exists. Each practical deliverable must be independently testable.

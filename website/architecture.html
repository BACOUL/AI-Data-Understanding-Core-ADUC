Composable by design

Architecture

ADUC is a small Core surrounded by reference tools, reusable vocabularies and extensions. It is designed to complement existing standards rather than replace them.

System view

Six principal components.

The specification defines the portable contract. The tools help produce, review and validate it. The registry and extensions add reusable knowledge without expanding the Core indefinitely.

1

Core Specification

Common blocks, identifiers, status rules, uncertainty, compatibility and consumer obligations.

2

Contract Format

A JSON or JSON-LD companion file that leaves the original source unchanged.

3

Compiler

Analyses JSON and CSV and proposes an inferred contract with evidence and confidence.

4

Validator

Checks schema, lifecycle, references, units, versions and local semantic conflicts.

5

Semantic Registry

References concepts, definitions, versions, owners, relations and compatible units.

6

Extensions

Add dataset, live-data, document, memory, scientific, situation and domain capabilities.

Layering

The original data remains the source of record.

ADUC does not force a migration to a new database or format. A contract accompanies the original resource and can be versioned independently.

Small Core

The Core defines only concepts that must remain portable across domains. It does not contain every medical, industrial, financial or scientific concept. Domain meaning belongs in external vocabularies and extensions.

Current repository architecture

The existing implementation mainly covers the semantics layer: mapping assertions, status, confidence, validation, comparison and RDF round-trip. ADR-0004 formally positions this work inside the full Core rather than deleting it or presenting it as complete.

Implemented foundation: semantic mapping schema, CLI validator, comparator, JSON-LD/RDF processing and conformance harness.
Next architecture work: full-Core schema modules for resource, structure, identity, context, provenance, uncertainty, relations and policy.

Extensions

Extensions must declare their own version and conformance identifier. A compatible extension may add properties or constraints, but it may not silently change the meaning of Core properties.

Near-term

Later

  • Document Extension

Project separation

TimeProofs may later timestamp releases, but it is not required to create or validate a contract. The anticipation engine may later consume ADUC, but it remains a separately governed application. This separation protects the credibility and portability of the Core.
